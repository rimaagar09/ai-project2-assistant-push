Intake & Input Types — SOW · New Requirement · Change Request · Product Brief

📑
Statement of Work (SOW)

Client-signed contract with defined deliverables, timelines, payment milestones.

Agent extracts: scope, deliverables, commercial terms, timeline, acceptance criteria, exclusions.

📋
New Project Requirement

Fresh requirement from business / product / client with no prior documentation.

Agent extracts: business problem, objectives, users/personas, modules needed, success criteria.

🔄
Change Request (CR)

Scope addition or modification to an existing live project or BRD.

Agent extracts: impacted sections, delta scope, effort estimate, risk to existing timeline.

💡
Product / Feature Brief

Internal brief from product team — feature launch, enhancement, or platform upgrade.

Agent extracts: feature spec, user flows, technical dependencies, rollout plan.

⚡ Intake Agent — What happens when input arrives

01
Classify input type

SOW → extract commercial + scope · Requirement → extract objectives + features · CR → extract delta scope + impact · Brief → extract spec + flows

02
Extract known answers

Agent reads the document and maps content to BRD sections. Every extractable answer is pre-filled. Agent only asks for what is missing.

03
Ask structured questions (gaps only)

5 question sets presented conversationally. Sets not applicable to the project type are skipped. No repeated or redundant questions.

04
Confirm extracted answers before writing

Agent presents a summary of what it understood — PM confirms or corrects. BRD is written only after confirmation.

🛡️ Guardrail Framework — auto-runs after every BRD · re-runs on every update
🛡️

7 Guardrail Categories — Key Gaps Report · Gap IDs · Owner · Due Date · Status Tracking

🔲G1 — Scope Boundaries
  • In/out-of-scope explicitly stated
  • Phase tags on every FR row
  • Change control process defined
  • Feature freeze date documented
  • No open TBDs without an owner
💰G2 — Financial Guardrails
  • Budget documented (range ok)
  • Contingency % defined (10–15%)
  • Cost overrun escalation threshold
  • Funding approval signatory named
  • Commercial model defined
⚖️G3 — Compliance & Regulatory
  • Applicable law assessed (GDPR / PDPA / PCI)
  • Consent / opt-in flows defined
  • PII & PSI data fields tagged
  • Data retention periods set
  • Right to erasure workflow defined
🎯G4 — SMART KPIs
  • ≥ 5 KPIs defined per module
  • Specific numeric targets
  • Time-bound horizon on each KPI
  • Baseline values documented
  • Measurement method defined
⛓️G5 — Business Constraints
  • Legacy system dependencies listed
  • Hard deadlines identified
  • Data migration scope defined
  • 3rd-party approval timelines noted
  • Cutover / parallel-run strategy set
👥G6 — Resource & Roles
  • RACI matrix defined
  • Decision authority documented
  • Escalation path defined
  • Sign-off owners per deliverable
  • Communication cadence set
🧪G7 — Test Case ExecutionNEW
  • Every FR traces to ≥1 test case (TC-xxx)
  • Critical workflows have negative + edge cases
  • Every NFR has a performance / security test
  • All test cases executed in UAT before prod
  • Failed tests logged as Jira defects
  • Defects resolved before go-live (or waived)

🗂️ Tab 4 — BRD Coverage Map All 19 sections ≥ partial

Visual heatmap of BRD section coverage. Red = missing · Yellow = partial · Green = complete. Gate 2 requires all sections ≥ partial.

🛡️ Tab 5 — Guardrail Status ← NEW Criticals must be 0

Live status of all 6 guardrail categories. Each gap: owner field + due date picker + status dropdown (Open / In Progress / Resolved / Waived). Gate 2 locked until all 🔴 resolved.

✅ Yes / No Quick Confirm — Worked Example

Is SMS required as a communication channel? (From BRD: "SMS listed as primary channel") ✅ YES — Confirmed
Is data migration from legacy CRM required? (From BRD: "existing CRM to be decommissioned") ✅ YES — Pre-filled
Is LINE integration required? (Not mentioned in BRD — needs input) ❓ Needs Confirmation
Is PCI-DSS compliance required? (Payment data mentioned but scope unclear) ❓ Needs Confirmation

What "100% Complete" means

  • All Must-Have questions answered (status ≠ Pending)
  • All Yes/No items confirmed — zero ❓ remaining
  • At least 1 module confirmed in Solution tab
  • BRD coverage map: all 19 sections ≥ partial
  • Guardrail tab: zero 🔴 Critical gaps open
  • Open Questions Register: all items have owner + due date

G7 — Test Case Execution: Traceability · Coverage · Test Types · Execution Governance

📐 Objective & Traceability

Why G7 exists

Every FR and NFR in the BRD must be validated by structured test cases before deployment. G7 enforces traceability, completeness, and quality assurance across the full implementation lifecycle — ensuring no requirement ships untested.

Re-runs after: BRD FR update · New UAT cycle · Defect resolution batch · Go-live checklist · User request "re-check G7".

🔬 Test Types + Execution Governance

6 Required Test Types

1
Functional Testing

Validates system behaviour vs BRD business requirements.

2
Integration Testing

Ensures correct communication between all connected systems.

3
Negative Testing

Tests invalid inputs, missing fields, and error conditions.

4
Edge Case Testing

Validates boundary conditions and extreme input values.

5
Regression Testing

Ensures existing functionality is unaffected by new changes.

6
UAT — User Acceptance Testing

Final validation by business stakeholders before go-live.

⚖️ Execution Governance Rules

  • All tests run in UAT env before production
  • All Critical + High priority TCs must pass
  • Failed TCs logged as Jira defects immediately
  • Defects resolved before go-live
  • Exceptions require PM + sponsor formal waiver
  • Go-live sign-off only after QA lead approval

Gap flags: 🔴 FR with no TC linked · 🔴 0% coverage on mandatory category · 🟡 UAT not signed off · 🟠 Evidence not attached to failed TC
